📩 SMS Format Requirements

The SMS must follow a specific format for the browser to extract the OTP code.

📱 SMS
Your verification code is 123456
@example.com #123456
Format rules:
  • Last line must be: @domain #code
  • @ prefix identifies the domain origin
  • # prefix identifies the OTP code
  • Domain must match the requesting page's origin
  • The rest of the SMS body can be any text

⏱️ AbortController Integration

Use AbortController to cancel the OTP listener after a timeout or user action.

No active timeout.
// With timeout const ac = new AbortController(); const timer = setTimeout(() => { ac.abort(); }, 60000); // 60 seconds try { const otp = await navigator.credentials.get({ otp: { transport: ['sms'] }, signal: ac.signal }); clearTimeout(timer); verifyCode(otp.code); } catch (err) { if (err.name === 'AbortError') { console.log('OTP request timed out'); } }

🛡️ Security Considerations

Origin-bound verification

The SMS must contain the requesting page's origin (@domain.com). The browser verifies this match, preventing phishing attacks from reading OTP codes meant for other sites.

User consent required

The browser shows a permission prompt before sharing the OTP code. The user must explicitly agree to share the code with the website.

SMS is not encrypted

SMS messages are transmitted in plaintext and can be intercepted. Web OTP does not make SMS-based 2FA more secure - it only improves the UX. Consider WebAuthn/passkeys for stronger security.

SIM swapping risk

If an attacker gains control of the phone number (SIM swap), they will receive the OTP. This is a fundamental SMS vulnerability, not specific to Web OTP.

Do not rely solely on SMS OTP

Use SMS OTP as one factor in multi-factor authentication, not as the sole verification method. Combine with knowledge factors (password) or possession factors (authenticator app).

AbortController for cleanup

Always use AbortController to cancel pending OTP requests when the user navigates away or the timeout expires, preventing resource leaks.
